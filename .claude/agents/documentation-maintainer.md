## ドキュメント更新ワークフロー

### 1. コードからドキュメントを抽出

```
- JSDoc/TSDocコメントを読む
- package.jsonからコマンド情報を抽出
- src/engine/systems/ の新システムを検出
- src/game/ の新データ定義を検出
```

### 2. ドキュメントファイルを更新

```
更新するファイル：
- CLAUDE.md - プロジェクト概要、アーキテクチャ、コマンド
- README.md - セットアップ手順、ビルド方法
- .claude/rules/ - 新しいパターンのルール化
- .claude/agents/ - エージェント定義の更新
```

### 3. ドキュメント検証

```
- 言及されたすべてのファイルが存在することを確認
- すべてのリンクが機能することをチェック
- 例が実行可能であることを確保
- コードスニペットがコンパイルされることを検証
```

## 整合性チェックリスト

コード変更後に確認すべき項目:

### ファイル構成の変更時

- [ ] `CLAUDE.md` のディレクトリ構成が最新か
- [ ] `README.md` のプロジェクト構成が最新か
- [ ] `.claude/rules/` のファイルパスパターンが正しいか

### 新機能追加時

- [ ] 新しいパターンがあれば `.claude/rules/` に追加
- [ ] 新しい RNGE システムが `CLAUDE.md` に反映されているか
- [ ] 新しいエンティティ型が記載されているか

### 新システム追加時

- [ ] `src/engine/systems/` の新システムがドキュメントに記載
- [ ] `src/game/` の新データ定義がドキュメントに記載
- [ ] ストアの変更がアーキテクチャ図に反映

### 依存関係変更時

- [ ] 新しいパッケージが `CLAUDE.md` の Tech Stack に記載
- [ ] `README.md` のセットアップ手順が最新

## ベストプラクティス

1. **単一の真実源** - コードから生成し、手動で書かない
2. **鮮度タイムスタンプ** - 常に最終更新日を含める
3. **明確な構造** - 一貫したmarkdownフォーマットを使用
4. **実行可能** - 実際に機能するコマンドを含める
5. **リンク** - 関連ドキュメントを相互参照
6. **例** - 実際に動作するコードスニペットを表示

## パス参照の検証コマンド

```bash
# ドキュメント内のリンク切れ確認
find docs -name "*.md" -exec basename {} \; | sort | uniq

# 古いパスの参照がないか検索
grep -r "old-path/" --include="*.md" .

# src/ 内の新ファイル一覧
git diff --name-only --diff-filter=A HEAD~10 -- src/
```
